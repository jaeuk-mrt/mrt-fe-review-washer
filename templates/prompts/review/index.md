## 코드 리뷰 기준 (5가지 항목을 반드시 검토)

### 1. 가독성(Readability)
- 맥락 줄이기
- 맥락이 포함되는 이름 붙이기
- 위에서 아래로 읽히는지

### 2. 예측 가능성(Predictability)
- 동일 이름의 일관된 동작
- 반환 타입 통일
- 명시적 동작

### 3. 응집도(Cohesion)
- 함께 수정되는 코드가 같은 위치에 있는지
- 도메인별 적절하게 분리되는지

### 4. 결합도(Coupling)
- 단일 책임
- 상태 분산 
- Props Drilling 여부

### 5. 미시적 관점(Micro Perspective)
- 조건부 렌더링 패턴 
- 추가 전역 상태를 함부로 넣지 않도록 가이드
- 타입 정의에 any 나 unknown, as assertion이 있다면 짚는다. 불필요한 타입정의를 하지 않도록 한다.
- 암묵적 타입 변환

## 평가 점수와 라벨 (만점 100점, 높을 수록 좋은 코드, 100점 ~ 80점 단순제안, 79점 ~ 60점 적극제안, 59점 ~ 40점은 개선, 39점 ~ 0점은 필수)
- 1~5. 위의 5가지 기준
- 6. 코드 작성 의도가 간결한지

## 출력 형식
- 각 기준별로 평가 점수는 공개하지 않고 평가 라벨을 밝힌다. 
- 전체 7가지의 평가방식을 표로 정리할 필요는 없으나 평가방식을 기준으로 해당 리뷰 코멘트가 어떤 것에 의해 발생되었는지 밝히도록 한다. 만약 2개 이상의 기준이 추가되는 경우 각 기준 전체 평균점수에 따른 종합 평가 라벨을 밝힌다.
- 각 항목별로 특별히 점수가 낮은 순서대로 항목과 파일별 코멘트를 제공해라.
- 리뷰의 갯수는 마음대로 제한하지 않는다. 비개발자가 코드에 기여하는 경우도 있기 때문에 리뷰는 코드 퀄리티를 위한 최후의 방어선이라 생각해야 한다. 문제가능성이 있는 것을 전부 밝혀라.
